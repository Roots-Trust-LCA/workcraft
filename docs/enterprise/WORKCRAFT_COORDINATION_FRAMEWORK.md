## Part I: Overview

### The Coordination Challenge

Organizations face a fundamental tension:

- **Centralize** → Efficiency and coordination, but loss of autonomy
- **Decentralize** → Resilience and autonomy, but fragmentation

World of Workcraft proposes a third way: **distributed coordination through explicit design**.

### What Is World of Workcraft?

WoWc provides organizational infrastructure that makes work legible:

1. **Clear Engagement Types** — Work classified by scope and coordination requirements
2. **Visible Capability Development** — Professional progression through demonstrated contribution
3. **Transparent Economic Flows** — Contribution tracking and value distribution
4. **Structured Cross-Functional Collaboration** — Role clarity without hierarchy

### Core Principles

| Principle | Description |
|-----------|-------------|
| **Node Autonomy** | Each operating unit governs itself completely |
| **Portable Identity** | Professional profile recognized across all nodes |
| **Shared Primitives** | Common infrastructure (credits, chain, patronage) |
| **Async-First Coordination** | Local-first, global when needed |
| **Data Sovereignty** | Each node holds its own data |

## Part II: Engagement Framework

### Engagement Types

Work is categorized by scope and coordination requirements:

| Type | Description | Coordination Level | Example |
|------|-------------|-------------------|---------|
| **Standing Commitment** | Routine, recurring work | Low | Operations, maintenance, heartbeat |
| **Process Improvement** | Non-critical enhancements | Low-Medium | Refactoring, documentation, tooling |
| **Strategic Initiative** | Core roadmap objectives | Medium-High | Product development, key deliverables |
| **Cross-Functional Initiative** | Multi-party coordinated work | High | Complex projects requiring role coordination |

### Engagement Structure

Each engagement includes:

1. **Scope Definition** — Clear objectives and deliverables
2. **Capacity Requirement** — Resource demand (time, compute, attention)
3. **Role Definitions** — Required functions and responsibilities
4. **Timeline** — Start, milestones, completion
5. **Allocation Framework** — How value is distributed upon completion

### Example: Cross-Functional Initiative

```
Initiative: Pollination Sensor Network

Scope: Deploy monitoring infrastructure across three watersheds

Capacity Required:
  - Infrastructure Lead: 40 hours
  - Systems Developer: 60 hours
  - Field Researcher: 30 hours

Timeline: 6 weeks
  - Week 1-2: Infrastructure setup
  - Week 3-4: Sensor deployment
  - Week 5-6: Data validation

Allocation:
  - Total: 200 credits + royalty share
  - Distribution: Based on contribution verification
  - Bonus: Early completion, quality metrics
```

## Part IV: Capacity Management

### The Capacity Budget System

**Capacity Budget** tracks available resources (time, compute, attention) for each professional.

**Key Concepts:**

1. **Budget Allocation** — Capacity budget per cycle (daily/weekly)
2. **Capacity Cost** — Each engagement has visible capacity requirement
3. **Utilization Tracking** — Real-time view of allocated vs. available capacity
4. **Recovery Period** — Renewal bonus after rest periods (prevents burnout)

### Capacity Dashboard

Each professional sees:

```
Professional: Maya
Role: Terraformer (Code × Earth)
Expertise: Living Systems Integration

Capacity Budget (Weekly):
  - Total: 100 units
  - Allocated: 65 units
  - Available: 35 units
  - Recovery Bonus: +10 units (after weekend rest)

Current Engagements:
  - Pollination Network (Cross-Functional): 40 units
  - Documentation Update (Process Improvement): 25 units

Upcoming:
  - Standards Review (Standing Commitment): 15 units
```

### Why Capacity Visibility Matters

**Before:** Professionals commit to work without seeing total load → burnout, missed deadlines

**After:** Capacity requirements visible before commitment → sustainable pacing, realistic planning

## Part V: Coordination Structures

### Four Layers of Coordination

| Layer | Scope | Purpose | Consent Required |
|-------|-------|---------|------------------|
| **Local Channels** | Within a node | Day-to-day coordination | Local governance |
| **Node Bridges** | Two nodes | Bilateral coordination | Both nodes |
| **Practice Communities** | Global (by function) | Standards, mentorship | Practice community |
| **Federation Council** | Ecosystem-wide | Shared infrastructure | All affected nodes |

### 1. Local Channels (Within a Node)

Each node organizes communication by seven layers:

```
Node: Techne (Boulder, Colorado)

├─ e/ Ecology
│  ├─ #watershed-updates
│  ├─ #seasonal-planning
│  └─ #local-projects
│
├─ H/ Human
│  ├─ #introductions
│  ├─ #practice-community-gatherings
│  └─ #member-announcements
│
├─ L/ Language
│  ├─ #terminology
│  ├─ #naming-debates
│  └─ #translation
│
├─ A/ Artifacts
│  ├─ #project-updates (by venture)
│  ├─ #resource-library
│  └─ #tools-and-templates
│
├─ M/ Methodology
│  ├─ #governance
│  ├─ #meeting-notes
│  └─ #decision-tracking
│
├─ T/ Training
│  ├─ #learning-circles
│  ├─ #skill-shares
│  └─ #knowledge-base
│
└─ S/ Sessions
   ├─ #upcoming-gatherings
   ├─ #convergence-prep
   └─ #event-notes
```

**Properties:**
- Fully local; no visibility to other nodes unless explicitly shared
- Asynchronous-first; threaded discussion, not real-time chat
- Moderated by node stewards who know the context
- Archive on local chain (immutable, verifiable)

### 2. Node Bridges (Between Two Nodes)

When two nodes coordinate, they form a **bridge**:

```
Bridge: Techne ←→ Nairobi Node

├─ #shared-ventures
│  ├─ Solar Cooperative (technical coordination)
│  └─ Patronage Research (cross-node learning)
│
├─ #knowledge-exchange
│  ├─ Ecological patterns: Africa vs. Americas
│  └─ Governance adaptations
│
└─ #economic-flows
   ├─ Cross-node credit transactions
   └─ Shared royalty tracking
```

**Forming a Bridge:**
1. Node A nominates a **bridge steward**
2. Node B accepts or proposes someone different
3. Together they publish a **covenant** describing purpose and decision flow
4. Bridge creates specific channels for coordination
5. Either node can dissolve the bridge at any time

**Properties:**
- Requires explicit consent from both nodes
- Decisions on bridge don't override local governance
- Bridge stewards are ambassadors, not authorities
- All activity mirrored to both nodes' local archives

### 3. Practice Communities (Global by Function)

Eight **Practice Communities** — global coordination for people practicing the same function:

```
Practice Community: Code Practitioners (all nodes)

├─ #infrastructure-standards
│  (coordinating data schema, chain versioning, credit mechanics)
│
├─ #craft-innovation
│  (sharing new patterns, libraries, tools)
│
├─ #expertise-mentoring
│  (experienced practitioners advising newcomers)
│
├─ #ecosystem-security
│  (coordinating vulnerabilities, incident response)
│
└─ #community-convergence
   (annual gathering of all Code practitioners)
```

**How Practice Communities Work:**
- Any member of the function can read and participate
- Decisions made by **consensus-seeking**, not voting
- Each community publishes **best practices** and **patterns**
- A node can adopt or adapt community standards
- If a node rejects a critical standard, ecosystem can apply economic pressure

**Community Officers** (one per node, elected by local members):
- Bridge between local practice and global coordination
- Represent their node's needs in community conversations
- No executive power — guidance and consensus-building only

### 4. Federation Council (Ecosystem-Level)

Coordinates infrastructure, economics, and governance affecting the whole ecosystem:

```
Federation Council

├─ Council Members
│  └─ One steward per node (rotated annually)
│
├─ Standing Committees
│  ├─ Patronage & Allocation
│  ├─ Chain & Cryptography
│  ├─ Federation Economics
│  ├─ Ecology & Bioregionalism
│  ├─ Learning & Knowledge Commons
│  └─ Conflict Resolution
│
└─ Decision Protocols
   ├─ Consensus: All affected nodes must agree
   ├─ Substantial Consensus: 80%+ nodes agree; dissenters can opt-out
   ├─ Simple Majority: 50%+ nodes agree; all must implement
   └─ One Node, Veto: Any node can veto decisions threatening autonomy
```

**What Federation Council Coordinates:**
- Credit mechanics and inflation/deflation
- Chain schema and cryptographic standards
- Federation infrastructure (shared servers, CDNs, archival)
- Major vulnerabilities and security responses
- Admission of new nodes
- Dissolution of nodes violating Federation Principles

**What Federation Council Cannot Do:**
- Override a node's local governance
- Require specific ventures or projects
- Set patronage formulas (those are local)
- Remove a member from the federation
- Dictate member expulsion from any node

## Part VI: Economic Infrastructure

### The Contribution Record

All work is recorded on an immutable chain:

- **What** — Engagement type, scope, deliverables
- **Who** — Contributors and roles
- **When** — Timestamp, cycle reference
- **Verification** — Attestations from peers or systems
- **Allocation** — Distribution upon completion

### Credit System

**Credits** enable internal value exchange:

- Earned through verified contributions
- Spent on services, resources, compute
- Bridges to patronage allocation
- Visible balance on professional profile

### Patronage Allocation

**Patronage** distributes value to contributors:

**Proposed Formula:**
- 40% Labor — Verified contributions (time, expertise)
- 30% Revenue — Value generated by ventures
- 20% Capital — Financial investment in infrastructure
- 10% Community — Governance participation, mentorship

## Part VII: Digital Infrastructure Trust

### The Challenge

When nodes rely on a single cloud provider (AWS, Google, etc.), they are not truly autonomous. The provider controls availability, pricing, data access.

### The Trust Model

**Digital Infrastructure Trust** — cooperative service holding and operating computing infrastructure for the federation.

**Who Owns It:**
- Any member of any federation node can join
- Governed as cooperative: one member, one vote
- Board: technical stewards from 3+ nodes (elected annually)

**What It Does:**
- Operates shared computing infrastructure
- Provides services to nodes on cost-recovery basis
- Maintains data residency and privacy guarantees
- Handles archival and disaster recovery
- Coordinates security updates and incident response

### Services Offered

| Service | Cost | SLA | Notes |
|---------|------|-----|-------|
| **Chain Archive** | Credit/byte/year | 99.9% uptime | Immutable storage |
| **Message Storage** | Credit/message/year | 99.9% uptime | Node archives, forums |
| **Compute** | Credit/core-hour | Per use case | Patronage calculations |
| **Transfer** | Credit/GB | Cost-based | Data movement |
| **Identity Service** | Credit/identity/year | 99.99% uptime | Profile resolution |

### Data Governance Principles

1. **Data Locality** — Each node's data held in its region
2. **Encryption by Default** — End-to-end encryption; nodes control keys
3. **Access Transparency** — Every access logged on chain
4. **Economic Efficiency** — Operates at cost; surplus returned as credits
5. **Exit Right** — Any node can leave with 60 days notice

## Part IX: Why This Matters

### The Core Problem

Most organizational coordination faces a hard choice:
- **Centralize** for efficiency and coordination
- **Decentralize** for autonomy and resilience

### The WoWc Solution

**Distributed coordination through explicit design.**

By making communication structures, professional development, and economic flows **legible and peer-to-peer**, we enable ecosystems that are:

- **Locally Rooted** — Each node thrives in its specific place
- **Globally Coordinated** — Nodes see each other and can collaborate
- **Economically Just** — Value flows based on contribution, not concentration
- **Resilient** — No single point of failure; exit is always an option
- **Emergent** — New forms of coordination arise from practice, not planning

### Professional Benefits

**For Individuals:**
- Visible capability development
- Portable professional identity
- Transparent economic participation
- Sustainable capacity management

**For Organizations:**
- Clear engagement classification
- Cross-functional coordination without hierarchy
- Retention through recognition and growth paths
- Economic alignment with contribution

**For the Ecosystem:**
- Resilient infrastructure (no central point of failure)
- Knowledge flow without extraction
- Economic justice built into the operating system
- Emergent coordination (standards arise from practice)
